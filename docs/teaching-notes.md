# Teaching Notes

## 学习偏好

- 像游戏闯关：提供简洁的全局路线图，但日常入口只展开当前一关，不把后续细节变成庞大待办。
- 按每周三次设计，不按每日打卡；首月预计 12 个微关。
- 每关约 10–20 分钟；忙碌时允许只做 2 分钟的“保温动作”。
- 先获得可见结果，再按需解释语法。
- AI 负责样板代码；学习者负责运行、观察、亲自改一处、用自己的话解释。
- 一个问题一个问题确认理解；答错表示关卡需要再切小，不表示失败。
- 不要求每日打卡，不补欠下的课程；下次从当前关继续。

## 难度调节

- 卡住 5 分钟：停止增加概念，把当前动作再拆成一半。
- 一次通过：仍只解锁下一关；可选挑战默认折叠。
- Boss 关只组合已学内容，不引入新知识；数据库 Boss 最多 30 分钟。
- 只有出现“运行结果 + 一处亲自修改 + 一句解释”才算真正通关。
- 首月完成 8、10 或 12 关分别代表基础、良好和目标通关，不要求补齐欠课。

## 螺旋式教学检查

- 通用方法以 [`learning-method.md`](learning-method.md) 为准；本文件只记录当前学习者的具体偏好。
- 每关提前写清楚一个旧锚点、一个主要新层次和一个明确延后点。
- 回忆不只指向上一关；每个阶段至少一次重新提取 2–3 关前的概念。
- 同一概念再次出现时，必须增加新场景、新操作、边界情况或理解深度。
- 学习记录只根据“能追踪、能解释、能修改、能排错”的实际证据更新，不用“已经讲过”代替掌握。
- 学习者的新疑问优先影响下一关的练习方式，固定阶段成果和技术边界保持稳定。

## 参考仓库取舍

- 保留：一次只打开一课、每课有产物、验证命令、检索问题、官方资料。
- 调整：不假设 Node.js/SQL 后端经验；把原来一天的内容拆成多个微关。
- 延后：slice 底层数组、rune、context、interface、gRPC、并发和可观测性。
- 改序：先建立 HTTP 的“代码 → 启动 → 请求 → 响应”回路，再按需要补 module、package 和语法。

## 后端主线

- 首月成果是 Go Todo REST API + MySQL，而不是完成一套 Go 语法目录。
- Prisma 暂不需要；首月也不引入 ORM、Redis、鉴权、事务和索引调优。
- 事务、索引、鉴权和 Redis 必须由真实场景触发，不能为凑技术清单提前加入。
- Docker 和数据库样板由 AI 主写，学习者只承担当前关需要理解与验证的部分。
