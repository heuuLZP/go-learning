持久化 Boss

上一关已经让列表查询和创建走 MySQL。本关把按 ID 查询、更新和删除也迁到数据库，移除内存数据，最后重启 Go 服务完成一次持久化 CRUD 验收。

本关胜利画面

通过 API 创建并更新一条 Todo，重启 Go 服务后仍能查询到更新结果；删除它后，API 返回 404 Not Found，MySQL 中也查不到这一行。

1. 先从动作推回 SQL

暂时不要看后面的代码，先说出三个 handler 各需要哪种 SQL：

GET    /todos/{id} → SELECT ... WHERE id = ?
PUT    /todos/{id} → UPDATE ... WHERE id = ?
DELETE /todos/{id} → DELETE ... WHERE id = ?

这里复用了第五到七关的路径 ID、第十关的 WHERE，以及上一关的参数占位符。问号对应的值仍然作为独立参数传给 database/sql，不要把 ID 或标题拼进 SQL 字符串。

2. 让单项查询读取数据库

把原来的 todoHandler 替换为接收 *sql.DB 的版本：

func todoHandler(db *sql.DB) http.HandlerFunc {
  return func(w http.ResponseWriter, r *http.Request) {
    id, err := strconv.Atoi(r.PathValue("id"))
    if err != nil {
      http.Error(w, "Todo ID 无效", http.StatusBadRequest)
      return
    }

    var todo Todo
    err = db.QueryRowContext(r.Context(),
      "SELECT id, title, done FROM todos WHERE id = ?", id,
    ).Scan(&todo.ID, &todo.Title, &todo.Done)
    if err == sql.ErrNoRows {
      http.Error(w, "Todo 不存在", http.StatusNotFound)
      return
    }
    if err != nil {
      http.Error(w, "查询 Todo 失败", http.StatusInternalServerError)
      return
    }

    w.Header().Set("Content-Type", "application/json")
    if err := json.NewEncoder(w).Encode(todo); err != nil {
      log.Printf("写入 JSON 响应失败: %v", err)
    }
  }
}

QueryRowContext 适合预期最多一行的查询。它把“没有结果”留到 Scan 时报告为 sql.ErrNoRows；handler 再把这个数据库结果翻译成客户端认识的 404。其他数据库错误只返回通用的 500，不把内部细节暴露给客户端。

3. 用 RowsAffected 判断更新目标

保留原来的 ID、JSON 和空标题检查，把查找并替换内存元素的部分改为参数化 UPDATE：

func updateTodoHandler(db *sql.DB) http.HandlerFunc {
  return func(w http.ResponseWriter, r *http.Request) {
    id, err := strconv.Atoi(r.PathValue("id"))
    if err != nil {
      http.Error(w, "Todo ID 无效", http.StatusBadRequest)
      return
    }

    var updatedTodo Todo
    if err := json.NewDecoder(r.Body).Decode(&updatedTodo); err != nil {
      http.Error(w, "请求 JSON 无效", http.StatusBadRequest)
      return
    }
    if strings.TrimSpace(updatedTodo.Title) == "" {
      http.Error(w, "title 不能为空", http.StatusBadRequest)
      return
    }

    result, err := db.ExecContext(r.Context(),
      "UPDATE todos SET title = ?, done = ? WHERE id = ?",
      updatedTodo.Title, updatedTodo.Done, id)
    if err != nil {
      http.Error(w, "更新 Todo 失败", http.StatusInternalServerError)
      return
    }

    affected, err := result.RowsAffected()
    if err != nil {
      http.Error(w, "读取更新结果失败", http.StatusInternalServerError)
      return
    }
    if affected == 0 {
      var existingID int
      err := db.QueryRowContext(r.Context(),
        "SELECT id FROM todos WHERE id = ?", id,
      ).Scan(&existingID)
      if err == sql.ErrNoRows {
        http.Error(w, "Todo 不存在", http.StatusNotFound)
        return
      }
      if err != nil {
        http.Error(w, "确认 Todo 失败", http.StatusInternalServerError)
        return
      }
    }

    updatedTodo.ID = id
    w.Header().Set("Content-Type", "application/json")
    if err := json.NewEncoder(w).Encode(updatedTodo); err != nil {
      log.Printf("写入 JSON 响应失败: %v", err)
    }
  }
}

ExecContext 只表示 SQL 已经执行，不表示这个 ID 一定存在。MySQL 默认把“写入的值和原值相同”也报告为影响 0 行，所以更新命中 0 行时要再查询一次：查不到才返回 404，查得到则是一次成功但没有产生变化的幂等更新。

4. 用同一种判断完成删除

删除与更新使用相同结构：解析 ID、执行带 WHERE 的 SQL、检查受影响行数。把原来的 deleteTodoHandler 替换为：

func deleteTodoHandler(db *sql.DB) http.HandlerFunc {
  return func(w http.ResponseWriter, r *http.Request) {
    id, err := strconv.Atoi(r.PathValue("id"))
    if err != nil {
      http.Error(w, "Todo ID 无效", http.StatusBadRequest)
      return
    }

    result, err := db.ExecContext(r.Context(),
      "DELETE FROM todos WHERE id = ?", id)
    if err != nil {
      http.Error(w, "删除 Todo 失败", http.StatusInternalServerError)
      return
    }

    affected, err := result.RowsAffected()
    if err != nil {
      http.Error(w, "读取删除结果失败", http.StatusInternalServerError)
      return
    }
    if affected == 0 {
      http.Error(w, "Todo 不存在", http.StatusNotFound)
      return
    }

    w.WriteHeader(http.StatusNoContent)
  }
}

成功时仍返回第七关学过的 204 No Content，不要再写 JSON 正文。执行两次相同的删除请求：第一次应为 204，第二次因为没有行可删，应为 404。

5. 接好路由并移除内存旁路

三个 handler 现在都需要数据库句柄，把路由注册改为：

http.HandleFunc("GET /todos/{id}", todoHandler(db))
http.HandleFunc("PUT /todos/{id}", updateTodoHandler(db))
http.HandleFunc("DELETE /todos/{id}", deleteTodoHandler(db))

确认五条 Todo 路由都已经接收 db 后，删除 main.go 中的内存 todos 切片和 nextTodoID。然后格式化并编译：

gofmt -w main.go
go test ./...

这里没有测试文件时，go test 仍会编译整个包。编译通过，才能继续最终验收。

6. 完成持久化 Boss

启动 MySQL 和 Go 服务，通过 POST 创建一条专用 Todo：

docker compose up -d
go run .
curl -i -X POST http://localhost:8080/todos \
  -H 'Content-Type: application/json' \
  -d '{"title":"level12 before restart","done":false}'

记下响应中的实际 ID。下面以 5 为例；如果你的 ID 不同，后续命令全部换成实际值。

curl -i http://localhost:8080/todos/5

curl -i -X PUT http://localhost:8080/todos/5 \
  -H 'Content-Type: application/json' \
  -d '{"title":"level12 after restart","done":true}'

现在按 Ctrl+C 只停止 Go 服务，不停止 MySQL，再执行 go run .。重新查询：

curl -i http://localhost:8080/todos/5

如果仍能看到更新后的标题和 "done":true，说明状态不在 Go 进程的内存里。再从 MySQL 交叉验证：

docker compose exec mysql \
  mysql -ugo_learning -pgo_learning go_learning \
  -e "SELECT id, title, done FROM todos WHERE id = 5;"

最后完成删除闭环：

curl -i -X DELETE http://localhost:8080/todos/5
curl -i http://localhost:8080/todos/5

两次响应应依次为 204 No Content 和 404 Not Found。再次执行上面的 MySQL 查询，应只剩表头，不再有这一行。

本关排错顺序

400 先检查路径 ID 或 JSON；404 先用 MySQL 查询实际 ID；500 再看 Go 终端和 MySQL 容器状态。先根据 HTTP 状态缩小范围，不要同时改路由、SQL 和数据库配置。
