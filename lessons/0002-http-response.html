看清一次 HTTP 响应

第一关已经让请求成功返回文字。这一关继续看清：状态码和正文怎样通过 w 发给客户端，以及 Fprint 与普通输出到底差在哪里。

本关胜利画面

GET /ping 返回 200 OK 和 pong；换成 POST 后，你能解释为什么会得到 405 Method Not Allowed。

1. 回忆上一关

先不看代码回答：请求匹配 GET /hello 后，哪一个函数负责写回响应？想好后再继续。

2. 认识新的 handler

func pingHandler(w http.ResponseWriter, r *http.Request) {
    w.WriteHeader(http.StatusOK)
    fmt.Fprint(w, "pong")
}

w 是这次 HTTP 响应的写入目标。WriteHeader 写入状态码，Fprint 把正文写入同一个响应。

顺序很重要：状态码要在正文之前写入。现在没有使用 r，后面的关卡会从中读取请求信息。

3. 看见状态码和正文

启动服务：

go run .

保留这个终端，再打开另一个终端运行：

curl -i http://localhost:8080/ping

在输出中分别找到：

HTTP/1.1 200 OK

pong

-i 让 curl 同时显示响应头。浏览器通常只直接展示正文，所以这一关改用 curl 观察完整响应。

4. 亲自比较输出位置

  1. 把 fmt.Fprint(w, "pong") 临时改成 fmt.Print("pong")。
  2. 停止并重新运行服务，再执行一次相同的 curl -i。
  3. 观察：pong 出现在运行 Go 服务的终端中，而 HTTP 响应只剩状态和响应头。
  4. 把代码恢复为 fmt.Fprint(w, "pong")，再次验证正文回来了。

fmt.Print 的默认目标是标准输出；fmt.Fprint 的第一个参数决定写到哪里。这里把 w 传进去，就是写入 HTTP 响应。

5. 请求方法也是路由条件

再发送一次 POST 请求：

curl -i -X POST http://localhost:8080/ping

这次会看到 405 Method Not Allowed。路径仍然是 /ping，但这条路由按 GET 方式注册，POST 不会匹配它。

6. 一题验收

把 fmt.Fprint(w, "pong") 换成 fmt.Print("pong") 后，为什么客户端看不到 pong？
