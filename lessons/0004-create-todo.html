用 POST 创建 Todo

上一关把 Go 数据编码成响应 JSON；这一关把方向反过来，从 HTTP 请求正文读取 JSON，创建一个新的 Todo。

本关胜利画面

发送 POST /todos 后看到 201 Created 和新 Todo；再次 GET /todos 时，列表中也能找到它。

1. 先看数据方向

上一关：todos → JSON 编码 → HTTP 响应
这一关：HTTP 请求 → JSON 解码 → todo → 追加到 todos

w 仍负责写响应；新出现的 r.Body 是客户端发来的请求正文。

2. 发出创建请求

启动服务：

go run .

在另一个终端发送 JSON：

curl -i -X POST http://localhost:8080/todos \
  -H 'Content-Type: application/json' \
  -d '{"title":"用 POST 创建 Todo","done":false}'

找到状态码和响应正文：

HTTP/1.1 201 Created
Content-Type: application/json

{"id":3,"title":"用 POST 创建 Todo","done":false}

201 Created 比 200 OK 更具体：它表示这次请求成功创建了一个新资源。

3. 沿着请求走进 Go

var todo Todo
json.NewDecoder(r.Body).Decode(&todo)

第一行先准备一个空的 Todo。第二行从 r.Body 读取 JSON，并把结果放进 todo。这里先把 &todo 理解为“把 todo 的位置交给 Decode”；指针语法以后再展开。

如果请求正文不是合法 JSON，handler 会返回 400 Bad Request，并停止创建。

4. 把新项放入列表

todo.ID = len(todos) + 1
todos = append(todos, todo)

服务端为新 Todo 分配 ID，append 再把它追加到 todos 列表。随后编码这个 todo，把创建结果写回响应。

5. 亲自改一个输入

  1. 把命令中的 title 改成你自己的文字，再发送一次 POST。
  2. 运行 curl -i http://localhost:8080/todos。
  3. 确认列表中出现你刚创建的 Todo，并对比两次请求的状态码和正文。

数据目前只保存在内存中，重启服务后会恢复为代码里写好的初始列表；持久化会在 MySQL 阶段解决。

6. 一题验收

json.NewDecoder(r.Body).Decode(&todo) 的数据从哪里来、到哪里去？
